## Plan: Credit Risk Model Monitoring & Governance Framework

### TL;DR
Design and document a complete, regulator‑aligned monitoring framework for credit risk models, covering data quality, drift, model performance, and governance. The plan is model‑agnostic (PD/LGD/EAD, retail/corporate) and tech‑agnostic, so it can be implemented in most environments; it emphasizes SR 11‑7‑style controls and clear triggers for action.

### Steps

**Phase 1 — Define Scope, Roles, and Governance**

1. Identify in‑scope models and uses — `model_inventory.xlsx` or risk system export  
   - List all credit risk models (PD, LGD, EAD, application/behavioral/collections, retail/corporate) and their primary uses (origination, limit management, provisioning, capital, collections, monitoring).  
   - For each model, record: owner, validator, business owner, deployment context (batch/real‑time), and jurisdictions/regulations applicable.

2. Classify model materiality and criticality  
   - Define simple tiers (e.g., High/Medium/Low) based on impact metrics: exposure size, regulatory capital/provisions reliance, decision criticality, customer impact.  
   - Assign each model to a tier; document tiering rules in a short policy note.

3. Define governance roles and committees  
   - Assign named individuals/teams to: Model Owner (analytics/risk), Business Owner, IT/Engineering Owner, Independent Validation, Model Risk Committee / Credit Risk Committee.  
   - Specify decision rights: who approves overlays, cut‑off changes, model redevelopments, and monitoring thresholds.

4. Draft model‑specific monitoring charters — `monitoring_plan_{model_name}.docx`  
   - For each material model, create a 3–5 page monitoring plan covering: intended use and limits of use, monitoring metrics (data, performance, business), frequency by tier (e.g., monthly for High), thresholds and actions, and reporting recipients.  
   - Include explicit SR 11‑7 linkage: conceptual soundness (not in scope here but reference validation), ongoing monitoring, and outcomes analysis.

5. Approve governance setup  
   - Socialize the generic monitoring framework and one example model plan in the Model Risk / Credit Risk Committee and obtain formal approval (minutes, sign‑off).  
   - Log approved monitoring plans into the model inventory.

**Verification for Phase 1**  
- Check: Every in‑scope model has a defined tier, named owner(s), and an approved monitoring plan stored in a central repository and referenced in the model inventory.

---

**Phase 2 — Define Metrics, Thresholds, and Triggers**

6. Specify data & process monitoring metrics  
   For each model, define:  
   - Data quality metrics: missingness rates, outlier counts, unexpected categories per feature; pipeline error rates.  
   - Data drift metrics: PSI for key features and score, alternative tests (KS/chi‑square/KL) where appropriate.  
   - Process integrity checks: version hashes of transformation code, reference data version IDs, count consistency before/after joins/filters.

7. Specify model performance metrics  
   - Discrimination: AUC/Gini, KS statistic, lift at relevant cut‑offs, overall and by key segments (product, region, channel, new vs. existing, secured vs. unsecured).  
   - Calibration: predicted vs. realized default rates for PD (calibration ratio, Brier score, calibration plot buckets), realized vs. predicted LGD/EAD (average error, MAPE, ratio).  
   - Stability: parameter stability if applicable (scorecards/regression coefficients on rolling windows) or feature importance stability for ML models.

8. Specify business & governance metrics  
   - Decision outcomes: approval/decline rates, average limit granted, pricing spreads, collections actions; stratified by segment.  
   - Business KPIs: default rate, NPL ratio, loss rate, risk‑adjusted return, provision accuracy for relevant horizons.  
   - Governance signals: override rates and reasons, use of manual exceptions, instances of temporary overlays.

9. Calibrate thresholds and color bands  
   - For each metric define: green/amber/red thresholds. Example: PSI < 0.1 (green), 0.1–0.25 (amber), >0.25 (red); Gini > 0.5 (green for consumer unsecured), 0.4–0.5 (amber), <0.4 (red) — adjust by portfolio.  
   - Use historical monitoring or development/out‑of‑time results to set realistic baseline thresholds; document rationale and any portfolio‑specific nuances.

10. Define explicit triggers and actions  
    - For each metric and tier, define what happens when amber/red thresholds are breached: additional analysis, heightened monitoring frequency, temporary overlays, tightening/loosening of cut‑offs, partial recalibration (e.g., PD mapping), or full redevelopment.  
    - Capture triggers/actions in a “Trigger‑Action Matrix” table inside each `monitoring_plan_{model_name}.docx`.

**Verification for Phase 2**  
- Check: For at least one pilot model, you can show a complete metric list with numeric thresholds and corresponding triggers, reviewed and accepted by Model Owner and Business Owner.

---

**Phase 3 — Data Capture and Logging Design**

11. Map required data elements for monitoring — `data_dictionary_monitoring.xlsx`  
    - For each model, identify all fields needed to compute metrics: input features, model scores, decisions (approve/decline, limits, prices), technical metadata (model version, code version), and eventual outcomes (defaults, losses, recoveries, utilization).  
    - Specify data types, allowed values, and source systems/tables for each field.

12. Design prediction logging schema — `monitoring_predictions_schema.sql`  
    - Define a canonical predictions log table (or set of tables) with mandatory columns:  
      - Technical: timestamp, model_name, model_version, environment, request_id.  
      - Input: all key features used by the model (or hashed/encoded if necessary for privacy).  
      - Output: score, PD/LGD/EAD estimates, decision, cut‑off used, override flag and reason if overridden.  
    - Ensure the schema supports both batch and real‑time prediction use cases.

13. Design outcomes/label capture schema — `monitoring_outcomes_schema.sql`  
    - Define how and where actual outcomes will be recorded: default flag and date, days‑past‑due buckets, write‑off date, recovered amount, exposure at default, utilization, etc.  
    - Include fields to link back to the original prediction record (e.g., request_id, account_id, snapshot_date).

14. Define data retention and sampling strategy  
    - Specify how long detailed logs will be retained (e.g., 7–10 years for regulatory purposes; shorter in high‑volume logs with aggregated archives).  
    - Define any sampling rules (e.g., keep 100% of high‑tier model logs; allow sub‑sampling only for low‑tier models) and aggregation rules for older data.

15. Align data privacy, security, and access control  
    - Classify sensitive attributes (e.g., personally identifiable information, protected characteristics) and determine whether they are stored, pseudonymized, or only used in fairness monitoring sandboxes.  
    - Define role‑based access: who can query raw logs vs. aggregated monitoring metrics.

**Verification for Phase 3**  
- Check: There is a complete data dictionary and ERD/diagram linking predictions logs and outcomes tables, reviewed by Data Engineering and Risk/Analytics, with documented agreement on retention and access rules.

---

**Phase 4 — Metric Computation & Monitoring Engine Design**

16. Choose monitoring computation environment and tooling  
    - Decide where metrics will be computed (e.g., scheduled batch jobs in data warehouse, Python/R jobs on a scheduler, vendor monitoring platform).  
    - Ensure chosen stack can handle joins between prediction and outcome data, calculates PSI, AUC, calibration statistics, and can segment results.

17. Define metric calculation specifications — `monitoring_metrics_spec_{model}.md`  
    - For each metric, document: formula, data required, time window (e.g., last month, last 3 months rolling), minimum sample size, segmentation hierarchy.  
    - Include details like binning rules for PSI and calibration plots (e.g., 10 score deciles or PD buckets).

18. Design batch monitoring jobs (sequential)  
    - Specify separate batch jobs for:  
      - Data quality & drift metrics (using latest prediction logs).  
      - Performance metrics (requiring prediction–outcome joins with appropriate lag windows).  
      - Business metrics (linking predictions to decision and portfolio data).  
    - Define job schedules per model tier (e.g., daily/weekly for high‑volume real‑time models, monthly/quarterly for low‑volume or slow‑realization models).

19. Design segmentation and drill‑down views  
    - Define the standard segmentation dimensions to be computed for each job (product, region, channel, customer type, vintage, secured/unsecured).  
    - Specify how the engine will output segment‑level metrics (e.g., one row per segment–metric–period in a fact table for dashboards).

20. Define storage format for monitoring results — `monitoring_metrics_schema.sql`  
    - Create schema for metrics tables, e.g.:  
      - `metric_name`, `model_name`, `model_version`, `segment`, `time_window_start`, `time_window_end`, `value`, `threshold_green`, `threshold_amber`, `threshold_red`, `band`.  
    - Include fields for comments and manually entered explanations for out‑of‑band metrics.

**Verification for Phase 4**  
- Check: For a static historical dataset (development + a hold‑out period), you can manually run your planned metric calculations (even in a notebook) and obtain a complete set of metrics and segment breakdowns consistent with the written specs.

---

**Phase 5 — Dashboards, Alerts, and Reporting Design**

21. Select visualization and alerting tools  
    - Decide on BI/monitoring stack (e.g., Power BI/Tableau/Looker vs. Grafana vs. vendor platform) and an alerting mechanism (e.g., email, Slack, ServiceNow tickets) that integrates with the metrics store.

22. Design standard monitoring dashboard layouts — `dashboard_wireframes_{model_or_portfolio}.pdf`  
    - Create wireframes for:  
      - Executive summary: top metrics, latest bands (green/amber/red), trend charts, and key exceptions.  
      - Data & drift tab: PSI charts for score and top features; data quality tables.  
      - Performance tab: ROC/AUC, KS, calibration plots, segment performance comparisons.  
      - Business impact tab: approval rates, loss rates, overlays, overrides, and KPIs over time.  
    - Ensure the ability to filter by model version, segment, and time window.

23. Define alert rules and routing — `alerting_rules.yaml`  
    - Translate trigger–action thresholds to alert rules specifying: metric, band, time aggregation, required persistence (e.g., 2 consecutive periods in amber), recipients, and severity.  
    - Decide which alerts open tickets and which only send notifications.

24. Define monitoring report templates — `monthly_monitoring_report_template.docx`  
    - Standardize narrative monitoring reports: executive summary, key metric tables and charts, explanation of breaches, actions taken, and decisions requested from committees.  
    - Include a section mapping outcomes to the triggers and showing the decision trail (e.g., overlays, re‑development proposals).

25. Integrate with governance calendar  
    - Map monitoring cycles (monthly/quarterly/annual) to existing committee calendars.  
    - Specify which dashboards/reports are reviewed in which forum (e.g., monthly risk operations meeting vs. quarterly Model Risk Committee).

**Verification for Phase 5**  
- Check: Stakeholders (Risk, Business, Model Risk) review sample dashboard mocks and report templates and confirm they contain all metrics they need, with clear visibility of thresholds and trends.

---

**Phase 6 — Implementation Roll‑out Plan (Process & Controls)**

26. Pilot implementation on one high‑priority model  
    - Select a representative, material model (e.g., retail application PD model) as the pilot.  
    - Apply all designs: create logs, compute metrics, populate dashboards, and run at least one full monitoring cycle using real or backfilled data.

27. Define operational run‑book — `monitoring_runbook_{model}.md`  
    - Document: job schedules, data dependencies, what to do when jobs fail, how to re‑run them, and contact points.  
    - Include a checklist for monthly/quarterly cycles: data checks, metrics computation, dashboard refresh, report drafting, and committee pack preparation.

28. Embed evidence and audit trail practices  
    - Define how monitoring evidence is archived (e.g., PDF exports of dashboards, signed reports, metric snapshots), with date/time stamps and model version info.  
    - Define change log practices: updates to monitoring thresholds, metric definitions, or dashboards must be tracked in a central change register and, where needed, approved.

29. Align with model validation and redevelopment processes  
    - Specify how monitoring results feed into periodic independent validation (e.g., if performance degraded, validation scope must address causes).  
    - Define hand‑off to model development teams when redevelopment triggers are hit (what documentation and data they receive).

30. Plan phased extension to all in‑scope models  
    - Prioritize remaining models based on tier and ease of integration, then create a timeline for adopting the framework across the portfolio.  
    - For each wave, assign accountable owners and target dates, and include them in the Model Risk Committee’s tracking.

**Verification for Phase 6**  
- Check: For the pilot model, there is a complete end‑to‑end chain in production or UAT: logs are populated, metrics are computed according to spec, dashboards and alerts work, and at least one monitoring report has been produced and reviewed in a governance meeting.

### Decisions

- **Model‑agnostic framework**: Chose a generic framework rather than tightly tailoring to a specific PD/LGD/EAD or portfolio type, because no model specifics were provided; this makes the design broadly reusable.  
- **Batch‑oriented monitoring**: Focused on batch/periodic monitoring as the default, with hooks for real‑time logging, because credit model monitoring typically relies on outcome realization over weeks/months, and regulators emphasize periodic reviews.  
- **PSI as primary drift metric**: Selected PSI as the standard drift measure, with optional statistical tests, because it is widely used and well understood in credit risk, and aligns with industry practices and vendor tools.

### Scope Exclusions

- Detailed technical implementation (code, pipelines, or specific vendor tool configuration) is out of scope; this plan defines what to implement and how it should behave, not the exact implementation.  
- Full stress testing and scenario analysis design is excluded; only the linkage to monitoring (e.g., using results to set overlays) is referenced.  
- Detailed fairness/bias monitoring across protected groups is not designed here; it is acknowledged as a risk but would require additional legal and data‑privacy input.

### Further Considerations

1. Extend the framework with fairness and disparate impact metrics where required by regulation or internal policy, integrating protected‑class analyses into monitoring dashboards.  
2. Incorporate automated challenger models and periodic re‑training pipelines (MLOps) that can be triggered when monitoring thresholds are breached.  
3. Align with enterprise‑wide model inventory and GRC tools to avoid duplication and ensure monitoring evidence directly supports internal and external audits.
