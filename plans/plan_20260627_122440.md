## Plan: Python Monitoring Package for Retail PD Origination Models

### TL;DR
Build a reusable, Python-based monitoring package that computes and reports data quality, stability, performance, and business-usage metrics for retail credit PD origination models. The core is a set of metric utilities, a standard monitoring dataset contract, and orchestrated monthly/quarterly runs that output tables (CSV/parquet) and plots (PNG/HTML) for inclusion in monitoring packs. Primary goal: clean, reusable, segmentable metrics with minimal assumptions beyond pandas and scikit-learn.

### Steps

**Phase 1 — Package Structure & Dataset Contract**

1. Create project skeleton — `credit_pd_monitoring/`  
   - `credit_pd_monitoring/__init__.py`  
   - `credit_pd_monitoring/data_prep.py`  
   - `credit_pd_monitoring/metrics_data_quality.py`  
   - `credit_pd_monitoring/metrics_stability.py`  
   - `credit_pd_monitoring/metrics_performance.py`  
   - `credit_pd_monitoring/metrics_business.py`  
   - `credit_pd_monitoring/reporting.py`  
   - `credit_pd_monitoring/config.py`  
   - `scripts/run_monthly_monitoring.py`  
   - `scripts/run_quarterly_monitoring.py`  
   - `tests/` (unit tests for each metric module)  
   - `pyproject.toml` or `setup.py`  
   - `README.md`

2. Define standard dataset schema & validation — `data_prep.py`  
   - Implement a `MonitoringSchema` class or `validate_input_df(df: pd.DataFrame)` that checks for:  
     - Required columns: `app_id`, `orig_date`, `product`, `pd_model`, `decision`, `override_flag`, `override_direction`, `default_12m`, `dpd30_3m`, `dpd30_6m`.  
     - Configurable segment fields: e.g. `country`, `segment_ntb_flag`, `channel`.  
     - Basic dtypes / coercions: parse `orig_date` to `datetime`, numeric cast for PD and outcomes.  
   - Allow configurable additional features list: `features_to_monitor: List[str]`.

3. Implement segmentation utilities — `data_prep.py`  
   - Implement helper functions:  
     - `add_standard_segments(df, config)` — ensures presence of `product`, `country`, `segment_ntb_flag`, plus any optional segment fields from config.  
     - `get_segment_slices(df, segment_by: List[str]) -> Dict[Tuple, pd.DataFrame]` to return grouped subsets by (product, country, etc.).  
   - Decide convention for vintage: e.g. `vintage_month = df['orig_date'].dt.to_period('M')`; add as a column.

4. Configure baseline datasets for PSI & development benchmarks — `config.py`  
   - Define a configuration object or YAML/JSON loader that stores:  
     - Filepath / identifier for baseline (development) dataset, or baseline distributions for key variables.  
     - Thresholds: PSI cutoffs, Gini/OE acceptable ranges.  
   - Provide a function `load_baseline_distributions()` that returns baseline series for PSI.

5. Verification (Phase 1)  
   - Create a tiny synthetic CSV (10–20 rows) in `tests/data/` that satisfies the schema.  
   - Write a unit test that:  
     - Loads the CSV into a DataFrame.  
     - Runs `validate_input_df` and `add_standard_segments` without errors.  
     - Confirms `vintage_month` and segment fields exist and have expected values.

**Phase 2 — Data Quality & Stability Metrics**

6. Implement missingness & basic quality metrics — `metrics_data_quality.py`  
   - Implement `data_quality_stats(df, vars_to_check, invalid_rules=None)` where `invalid_rules` is a dict mapping variable → callable `(series -> boolean mask)` for invalids.  
   - Output columns: `variable`, `missing_rate`, `invalid_rate` (if rules provided), `n_total`.  
   - Ensure output is a tidy DataFrame, easy to pivot by `vintage_month` and segments.

7. Implement PSI calculation — `metrics_stability.py`  
   - Implement `calculate_psi(expected: pd.Series, actual: pd.Series, bins=10)` as per the sketch.  
   - Add higher-level function `psi_for_variables(baseline_df, current_df, vars_to_check, bins=10)` that:  
     - Ensures identical binning across baseline/current.  
     - Returns DataFrame: `variable`, `psi`, `baseline_period`, `current_period`.  
   - Consider fallback handling for categorical variables (optional, but at least support numeric first, and raise clear error for non-numeric).

8. Integrate PSI against baseline configuration — `metrics_stability.py`  
   - Implement `compute_all_psi(current_df, baseline_df, key_vars, config)` that:  
     - Uses baseline period defined in `config`.  
     - Applies PSI to:  
       - `pd_model` or `score`  
       - Top N features (configurable list).  
   - Tag results with `vintage_month` and segments.

9. Verification (Phase 2)  
   - Create test where current_df has a clear distribution shift (e.g., add constant to `bureau_score`): expect PSI > 0.25.  
   - Test missing/invalid calculation with NaNs and deliberately invalid ages/incomes; assert rates match hand-calculated values.

**Phase 3 — Model Performance Metrics**

10. Implement discrimination metrics — `metrics_performance.py`  
    - Functions:  
      - `compute_auc(df, y_col='default_12m', pd_col='pd_model')` (as in sketch).  
      - `compute_gini(df, y_col='default_12m', pd_col='pd_model')`.  
      - `ks_statistic(df, y_col='default_12m', score_col='pd_model')`.  
    - Implement wrapper `discrimination_metrics(df, y_col, pd_col)` returning dict/DataFrame with AUC, Gini, KS, and count of defaults/non-defaults.

11. Implement calibration metrics — `metrics_performance.py`  
    - Overall O/E: `oe_ratio(df, y_col, pd_col)`.  
    - Bucket-level calibration: `bucket_calibration(df, y_col, pd_col, bins)` (sketch already provided).  
    - Optional Brier: `compute_brier(df, y_col, pd_col)`.  
    - Implement `full_calibration_summary(df, y_col, pd_col, bins)` returning:  
      - Overall OE, Brier.  
      - Per-band `avg_pd`, `obs_dr`, `n`, and optionally `oe_band = obs_dr / avg_pd` where `avg_pd > 0`.

12. Implement early-arrears metrics — `metrics_performance.py`  
    - `early_arrears_rate(df, arrears_col='dpd30_3m')` (sketch).  
    - `early_arrears_by_band(df, pd_col, arrears_col, bins)` similar to bucket calibration but with early arrears instead of default_12m.  
    - Provide mapping from vintage to available outcomes: e.g., function that filters to vintages with at least 3/6/12 months elapsed, if `as_of_date` is available (optional but recommended).

13. Add segmentation-aware runners — `metrics_performance.py`  
    - Implement `performance_by_segment(df, segments, y_col, pd_col, bins)` that:  
      - Groups by `[vintage_month] + segments`.  
      - For each group, computes discrimination metrics and OE.  
      - Returns a single concatenated DataFrame with columns: `vintage_month`, segments, `auc`, `gini`, `ks`, `oe`, `brier`, `n`, `n_defaults`.

14. Verification (Phase 3)  
    - Use synthetic data where PDs are perfectly rank-ordered with outcomes → expect AUC ≈ 1, high Gini, large KS.  
    - Create a case where PDs equal the observed default rate → OE ≈ 1 and Brier minimal compared to a distorted PD version.  
    - Confirm bucket calibration returns plausible band-level stats (monotonic PD vs. observed DR under controlled synthetic setup).

**Phase 4 — Business & Usage Metrics**

15. Implement decision metrics — `metrics_business.py`  
    - `decision_metrics(df, pd_col='pd_model')` as per sketch.  
    - `decision_metrics_by_segment(df, segments, pd_col)` that:  
      - Groups by `vintage_month + segments`.  
      - For each group computes: `approval_rate`, `avg_pd_approved`, `avg_pd_declined`, `n_total`, `n_approved`, `n_declined`.

16. Implement override metrics — `metrics_business.py`  
    - `override_stats(df)` as per sketch.  
    - `override_stats_by_segment(df, segments)` that computes:  
      - Overall override rate.  
      - Breakdown by `override_direction` share.  
      - Optionally, mean PD of overrides by direction.

17. Optional: risk appetite alignment checks — `metrics_business.py` *(parallel)*  
    - If configuration provides target max average PD by segment, implement:  
      - `check_risk_appetite(df, segments, pd_col, targets_config)` returning flags where average approved PD exceeds target.  
    - This step can be stubbed if targets are not known; leave hooks in place.

18. Verification (Phase 4)  
    - Build small test dataset where approvals and declines have clearly different PD distributions: assert `avg_pd_declined > avg_pd_approved`.  
    - Create overrides with known counts; verify override rate and direction shares match.

**Phase 5 — Reporting & Orchestration**

19. Design output structure — `reporting.py`  
    - Choose standardized output folders (relative to run date):  
      - `outputs/{run_date}/tables/`  
      - `outputs/{run_date}/plots/`  
    - Define reporting data contracts:  
      - Tables: CSV/Parquet for each metric family, e.g., `data_quality_by_segment.csv`, `psi_summary.csv`, `performance_by_segment.csv`, `decision_metrics_by_segment.csv`, `override_stats_by_segment.csv`.  
      - Plots (optional at first): ROC curves, calibration plots, PSI bar charts per variable, OE over time.

20. Implement table generators — `reporting.py`  
    - `generate_monthly_tables(df, baseline_df, config)` performing:  
      - Data quality stats for monitored variables by `vintage_month` and core segments.  
      - PSI for `pd_model`/top drivers vs. baseline.  
      - Decision metrics and early arrears where available.  
      - Override stats.  
      - Save each as CSV into `tables/`.  
    - `generate_quarterly_tables(df, baseline_df, config)` performing:  
      - Discrimination metrics for vintages with ≥ 12-month outcome.  
      - Calibration (OE, bucket-level).  
      - PSI trends.  
      - Overrides breakdown.  
      - Save each as CSV.

21. Implement core plots — `reporting.py` *(parallel)*  
    - Use matplotlib or seaborn; functions such as:  
      - `plot_calibration(grouped_calibration_df, output_path)` — line/point plot of `avg_pd` vs `obs_dr`.  
      - `plot_psi_bar(psi_df, output_path)` — bar chart of PSI per variable with thresholds annotated.  
      - `plot_oe_trend(oe_by_period_df, output_path)` — line plot of OE over vintages.  
    - Save to `plots/` as PNG.

22. Build orchestration scripts — `scripts/run_monthly_monitoring.py` and `scripts/run_quarterly_monitoring.py`  
    - Monthly script should:  
      - Parse CLI args/config (e.g., input data path, baseline data path, run_date).  
      - Load DataFrame; run `validate_input_df` and `add_standard_segments`.  
      - Load baseline; call `generate_monthly_tables` (and optionally plots).  
    - Quarterly script should:  
      - Do the same, but call `generate_quarterly_tables`.  
      - Optionally restrict to vintages with full 12-month outcome using `orig_date` vs `as_of_date`.

23. Documentation — `README.md`  
    - Document:  
      - Required input columns and acceptable formats.  
      - How to configure monitored variables and baseline.  
      - How to run monthly/quarterly scripts with examples.  
      - High-level explanation of metrics and thresholds (linking back to your metric dictionary).

24. Verification (Phase 5)  
    - Run monthly script on the synthetic dataset; confirm:  
      - All expected CSV files are produced.  
      - Content columns/row counts align with expectations.  
    - If plots implemented, open a couple manually and verify they reflect underlying tables.  
    - Run quarterly script on synthetic data with simulated 12-month outcomes to confirm end-to-end performance metrics.

### Decisions

- **Python package + scripts vs. single notebook**: Chose a reusable package with CLI scripts rather than a monolithic notebook, so the same codebase can support multiple models, automated scheduling, and unit tests.  
- **Pandas + scikit-learn vs. heavier workflow frameworks**: Limited dependencies to pandas, NumPy, and scikit-learn (plus matplotlib/seaborn for plots) to keep the monitoring lightweight and easy to deploy in most environments, avoiding airflow/Prefect-level orchestration as out-of-scope.  
- **Config-driven baselines & segments**: Chose a configuration-based approach (config file/module) for specifying baseline data and segment definitions so the same code can be reused across products, countries, and future models without code edits.

### Scope Exclusions

- Automated model recalibration or overlays: The monitoring will flag mis-calibration (e.g., OE outside bands) but will not implement recalibration logic; that remains a separate governance process.  
- Real-time/streaming monitoring: The implementation assumes batch (monthly/quarterly) runs on static datasets, not streaming Kafka/real-time monitoring pipelines.  
- Fairness/bias and regulatory explainability metrics: Out-of-scope for this iteration; the focus is on technical performance, stability, and business usage as defined in the provided metric dictionary.

### Further Considerations

1. Add support for categorical PSI (based on proportions per category instead of numeric binning) and automatically infer variable type.  
2. Introduce a simple HTML report generator (e.g., Jinja2 templates) to bundle tables and plots into a ready-to-share monitoring pack.  
3. Extend the framework to cover IFRS 9/behavioral PDs (e.g., stage migrations, transition matrices) and LGD/EAD metrics in a future phase.
