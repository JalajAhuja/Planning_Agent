## Plan: Basel/IFRS‑Compliant Monitoring Framework for Retail Origination PD

### TL;DR
Design and roll out a regulator‑ready monitoring framework for a retail origination PD model, using current on‑prem / spreadsheet‑heavy tooling but structured so it can be industrialized and reused across models. The plan covers governance, metric definitions, data flows, scripts, templates, thresholds, and evidence for audit and supervisors.

### Steps

**Phase 1 — Governance, Scope & Metric Dictionary**

1. Draft model‑specific Monitoring Framework document — `docs/PD_Origination_Monitoring_Framework.docx`  
   - Define model covered (name, version, implementation date, scope: products, countries, segments).  
   - Specify objectives (discrimination, calibration, stability, appropriate use, IFRS 9 linkage).  
   - Describe roles: Model Owner, Business Owner, Validation, Data/IT, Committees.  
   - Embed monitoring frequencies (monthly/quarterly/annual) and escalation principles.
2. Define metric dictionary and thresholds — `docs/PD_Origination_Metric_Dictionary.xlsx`  
   - List all metrics by layer:  
     - Data/process: record counts, missingness, invalid values, duplicates, PSI per variable and score.  
     - Performance: AUC/Gini, KS, lift, O/E, PD band calibration, Brier or similar.  
     - Business/use: approval rates, booked PD distribution, 30+ dpd within 3/6 months, overrides.  
   - For each metric, specify: definition/formula, slice dimensions (e.g., product, geography, new‑to‑bank vs. existing, channel), minimum sample sizes, thresholds, and breach actions.  
   - Document regulatory rationale per metric (Basel/EBA/ECB expectations, IFRS 9 requirement where relevant).
3. Define segmentation and scope of analysis — `docs/Segmentation_Spec_PD_Origination.docx`  
   - Fix primary segments: product type, country/region, new‑to‑bank vs. existing, main channels (branch/online/partner).  
   - Define any additional business‑critical cuts (e.g., income bands, collateral vs. unsecured).  
   - Describe when a segment is exempt from some metrics (e.g., too few defaults → skip Gini or provide only indicative).
4. Set governance thresholds and escalation workflow — section in `PD_Origination_Monitoring_Framework.docx`  
   - Convert metric thresholds into explicit rules, e.g.:  
     - PSI ≥ 0.25 on score distribution or key features → raise to Model Owner and Data/IT; analyze cause within X working days.  
     - Gini drop > 10 points from development or below absolute floor → trigger performance review.  
     - Overall O/E outside [0.8, 1.25] or systematic mis‑calibration in 3+ contiguous PD bands → propose recalibration or overlays.  
   - Define decision routes: which breaches go to which committee (e.g., Retail Credit Committee vs. Model Risk Committee).
5. Obtain formal approval of monitoring framework — internal model governance  
   - Circulate documents to Model Owner, Business Owner, Validation, and Model Risk.  
   - Capture sign‑offs and effective date in a simple approval log — `docs/Monitoring_Framework_Approvals.xlsx`.

**Phase 2 — Monitoring Dataset Design & Data Extraction**

6. Design monitoring dataset schema — `docs/Monitoring_Data_Model_PD_Origination.xlsx`  
   - Define unified schema (one row per scored application):  
     - Identifiers: application ID, customer ID (if any), origination date, decision date.  
     - Portfolio attributes: product, geography, channel, new‑to‑bank vs. existing.  
     - Model inputs: all features or at least top 10–20 drivers, with clear names and types.  
     - Model outputs: score, PD, segment/bucket, cut‑off flags.  
     - Decision data: approve/decline, override flag, override direction (upgrade/downgrade), override reason code.  
     - Outcomes: days‑past‑due buckets by month, delinquency flags (30+, 60+, 90+ dpd), default flag and date, write‑off.  
   - Document source systems and tables for each field (loan origination, core banking, collections, data warehouse, etc.).  
   - Specify vintage definition (e.g., based on origination month).
7. Implement initial manual extraction process — `sql/monitoring_pd_origination_extract.sql`  
   - Write SQL queries (or equivalent ETL spec) that:  
     - Pull all scored applications for a given calendar month, with all schema fields.  
     - Join outcome and delinquency status for all possible months since origination.  
   - Parameterize by extract month; output to a controlled location as CSV or table (e.g., `MON_PD_ORIG_VINTAGE_YYYYMM`).  
   - Ensure reproducibility: script checked into version control, run parameters logged.
8. Define and populate a “golden record” test sample — `data/golden_sample_PD_Origination.csv`  
   - Identify a small, stable test set (e.g., 100–500 applications across typical segments) scored with current production model.  
   - Record all input features and expected model score/PD at the time of initial creation.  
   - Store under strict change control; changes only allowed under formal governance.
9. Document data lineage and controls — `docs/Data_Lineage_PD_Origination_Monitoring.docx`  
   - For each field in the monitoring dataset, specify: source, key joins, any transformations.  
   - Describe how monthly extracts are triggered, where files are stored, how completeness and reconciliation to core systems are checked (e.g., total applications vs. origination reports).

**Phase 3 — Metric Computation Scripts & Validation**

10. Implement core monitoring functions — `code/metrics_pd_monitoring.R` or `.py`  
    - Functions for:  
      - Data quality: missing rate, invalid value counts, duplicate application IDs, record counts by segment.  
      - PSI: for each chosen feature and the score, with reference distribution from development or baseline period.  
      - Discrimination: AUC/Gini, KS, lift tables, by vintage and segment.  
      - Calibration: O/E ratio overall, PD band curves, Brier score or simple alternative.  
      - Business metrics: approval rate, booked PD distribution, early arrears (30+ dpd at 3 and 6 months), override rates and patterns.  
    - Ensure each function takes standardized inputs (dataframe, variable names, optional segment variable) and returns tidy tables suitable for export (CSV/Excel).
11. Implement “golden record” regression test script — `code/golden_regression_test_PD_Origination.R` or `.py`  
    - Load `data/golden_sample_PD_Origination.csv`, call production scoring routine, and compare output scores/PDs to stored expected values.  
    - Define pass/fail rules (e.g., absolute difference < small tolerance or exact match).  
    - Output a small summary file/log for inclusion in monthly monitoring evidence.
12. Build monthly monitoring computation driver — `code/run_monthly_PD_Origination_monitoring.R` or `.py`  
    - Steps:  
      - Read the monthly monitoring dataset for the target vintage.  
      - Run data quality checks and produce tables by key dimensions.  
      - Compute PSI for top features and score vs. reference distribution (stored baseline or development sample).  
      - For vintages with sufficient outcomes: compute discrimination and calibration metrics by segment.  
      - Compute business/use metrics (approval rate, overrides, early arrears).  
      - Save all metric outputs as structured CSV/Excel files in e.g. `output/PD_Origination/YYYYMM/`.  
    - Include run metadata (vintage, code version, run time) in an index file for audit.
13. Validate metric calculations using development sample — one‑off check  
    - Apply scripts to the original development dataset (or a close proxy).  
    - Confirm that computed: Gini, KS, PSI (vs. itself), and calibration all match or are consistent with what was reported in the original model documentation.  
    - Document any acceptable differences and their reason (e.g., binning choices) — `docs/Metric_Validation_Results_PD_Origination.docx`.

**Phase 4 — Reporting Templates & Review Workflow**

14. Create standard monitoring pack template — `reports/templates/PD_Origination_Monthly_Monitoring_Pack.pptx`  
    - Sections:  
      - Executive summary: key movements vs. prior month and vs. development, threshold breaches.  
      - Data & process: record counts, missingness trends, PSI charts (features + score), golden record test status.  
      - Model performance: Gini/KS/lift trends by vintage and key segments; calibration curves; O/E ratios.  
      - Business outcomes: approval and decline metrics, booked risk profile, early arrears rates, Stage 1/Stage 2 mix for new accounts.  
      - Overrides and behavior: override rates, top branches/segments with high overrides, qualitative comments.  
      - Actions & follow‑up: open issues, implemented actions, owner and target date.  
    - Use chart and table placeholders that can be populated from standardized CSV/Excel outputs.
15. Build Excel aggregation file for report feeding — `reports/templates/PD_Origination_Monthly_Data_For_Reporting.xlsx`  
    - Define tabs aligned with pack sections (DataQuality, PSI, Gini, Calibration, BusinessMetrics, Overrides).  
    - Design formulas or pivot tables linked to the outputs from Step 12, so users can refresh each month by simply updating data sources.  
    - Where possible, define named ranges corresponding to chart elements in the PowerPoint template.
16. Define monthly and quarterly review calendar — `docs/Monitoring_Review_Calendar_PD_Origination.xlsx`  
    - Specify:  
      - Cut‑off date for data inclusion (e.g., last business day of month).  
      - Due date for running scripts and producing packs.  
      - Standing meetings:  
        - Monthly PD Monitoring Review (Risk + Business) — review data quality, PSI, short‑term outcomes and business metrics.  
        - Quarterly Model Performance Review (add Validation) — review discrimination, calibration, overrides, and back‑testing.  
    - Clarify who prepares the pack, who presents, and who signs off.
17. Implement issues and actions log — `logs/PD_Origination_Monitoring_Actions.xlsx`  
    - Columns: date, metric/area, breach or observation, root‑cause analysis summary, agreed action, owner, due date, status, and closure date.  
    - Make this log a standing appendix to the monitoring pack.

**Phase 5 — Semi‑Automation & Control Environment**

18. Schedule monthly data extraction job — ETL or scheduler (e.g., cron/Control‑M)  
    - Wrap `sql/monitoring_pd_origination_extract.sql` (or equivalent ETL) in a scheduled job, parameterized by month.  
    - Configure output into designated monitoring area with predictable filename format.  
    - Implement basic controls: job completion alert; record counts logged vs. previous runs.
19. Schedule monthly metric computation — same scheduler  
    - After successful data extraction, trigger `code/run_monthly_PD_Origination_monitoring.R` or `.py`.  
    - Log runtime, counts, and key metric summary (e.g., Gini, overall PSI, O/E, early arrears) to a simple text or CSV summary.  
    - On failure, send email/notification to Model Owner and Data/IT contacts.
20. Integrate golden record regression test into monthly run  
    - Execute `code/golden_regression_test_PD_Origination.R` or `.py` as part of the same job chain.  
    - Abort or flag monitoring run if the golden record test fails; document the incident in the actions log and treat as potential model implementation change.
21. Introduce basic alerting for threshold breaches — `code/evaluate_thresholds_PD_Origination.R` or `.py`  
    - Parse latest metrics outputs and compare against thresholds defined in the metric dictionary.  
    - For each breach, auto‑create a short machine‑readable summary (metric, value, threshold, segment) and email to Model Owner with a standard template.  
    - Optionally, populate a simple database or shared file for subsequent aggregation of breaches.

**Phase 6 — Documentation, Back‑Testing & Extension Preparation**

22. Compile model‑specific Monitoring Plan document — `docs/PD_Origination_Monitoring_Plan.docx`  
    - Integrate: monitoring objectives, metric dictionary, segmentation, data model, scripts, templates, thresholds, and workflow.  
    - Include process flowcharts (data extraction → metrics → pack → committee).  
    - Map content explicitly to Basel/EBA/ECB and IFRS 9 expectations (e.g., references to internal policy paragraphs).
23. Establish report and data archiving process  
    - Define folder conventions:  
      - `archive/PD_Origination/reports/YYYY/MM/` for packs and raw outputs.  
      - `archive/PD_Origination/data/YYYY/MM/` for monitoring datasets.  
    - Specify retention period (e.g., at least 7 years or per regulatory requirement).  
    - Restrict access appropriately and document in `docs/Monitoring_Archiving_Policy_Excerpt_PD_Origination.docx`.
24. Design annual back‑testing analysis — `docs/PD_Origination_Annual_Backtesting_Template.docx`  
    - Outline analyses:  
      - Full‑period discrimination and calibration by product and geography.  
      - Benchmark comparison to a simple challenger (e.g., bureau‑only model).  
      - Review of overrides patterns, action effectiveness, and any overlays.  
      - Recommendation (keep, recalibrate, rebuild, or retire model).  
    - Define data requirements (e.g., all vintages with ≥12 months outcomes) and roles (who prepares vs. who reviews).
25. Prepare blueprint for extension to other models — `docs/Enterprise_Monitoring_Framework_Blueprint.docx`  
    - Generalize components: metric modules, dataset schema patterns, governance templates.  
    - Identify which parts are generic (PSI, Gini, calibration, overrides analysis) and which are PD‑type specific (e.g., 12‑month horizon, origination vintage definition).  
    - List priority candidate models for next rollout (e.g., behavioural PD, LGD, IFRS 9 lifetime PD).

### Verification

1. Governance & Design (Phase 1–2)  
   - Check that `PD_Origination_Monitoring_Framework.docx`, `Metric_Dictionary.xlsx`, and `Monitoring_Data_Model.xlsx` are fully populated and signed off by Model Owner, Business Owner, and Validation.  
   - Confirm segmentation and metric thresholds in the documents match internal risk policies.
2. Data Extraction & Quality (Phase 2)  
   - Run `monitoring_pd_origination_extract.sql` for at least two months and:  
     - Reconcile record counts against origination system reports by portfolio/segment.  
     - Verify all required fields are present and populated within expected ranges.  
   - Review a small random sample manually against source systems for correctness.
3. Metric Scripts (Phase 3)  
   - Execute `run_monthly_PD_Origination_monitoring.R` / `.py` on a known vintage:  
     - Confirm outputs are generated for all specified metrics and segments.  
     - Compare results against independently computed samples (e.g., using SAS or Excel) to validate calculations.  
   - Run `golden_regression_test_PD_Origination.R` / `.py` and confirm PASS status with zero or acceptable mismatches.
4. Reporting & Workflow (Phase 4)  
   - Generate one full monthly pack using the templates, based on recent data.  
   - Present it in a trial review meeting with Risk and Business; confirm that:  
     - All required sections are filled.  
     - Threshold breaches are clearly indicated.  
     - Action points are captured in `Monitoring_Actions.xlsx`.
5. Automation & Controls (Phase 5)  
   - Trigger the scheduled job chain for one full cycle (extraction → metrics → golden test → threshold evaluation).  
   - Verify: logs are created, notification emails are sent in case of test or threshold failures, and files land in correct directories.  
   - Check that failed runs are detectable and re‑runnable without manual data tampering.
6. Documentation & Audit Readiness (Phase 6)  
   - Ensure that for at least three consecutive months, monitoring packs, raw metric outputs, and datasets are archived per the defined structure.  
   - Perform a mock audit review: using only archived artefacts and documents, test that an independent reviewer can understand processes, reproduce a sample metric, and trace an identified issue to actions taken.

### Decisions

- **Separate monitoring dataset vs. direct use of operational tables**: Chose a dedicated monitoring dataset schema to ensure stability, reproducibility, and clear lineage, and to decouple analysis from operational schema changes. Directly querying live operational tables each month was rejected due to higher risk of breakage and audit complexity.  
- **Script‑based metrics with spreadsheet reporting vs. fully manual Excel**: Implemented metrics in R/Python and used Excel/PowerPoint only for visualization. A fully manual Excel‑only approach was rejected because it does not scale, is error‑prone, and is harder to audit; a full BI/MLOps platform was deferred to a later phase due to current on‑prem, spreadsheet‑heavy environment.  
- **Static golden sample regression test vs. ad‑hoc checks**: Adopted a formal golden record approach as a consistent, low‑cost control on the score implementation. Relying only on ad‑hoc sanity checks or occasional parallel runs was rejected because supervisors and internal model risk functions generally expect a clear, repeatable implementation control.

### Scope Exclusions

- Real‑time or near‑real‑time monitoring of the PD model (e.g., streaming dashboards) — excluded because current architecture is batch/on‑prem and regulators do not require real‑time for this use case.  
- Automatic model retraining or self‑adjusting calibration in production — excluded to avoid conflicts with model governance; any recalibration or redevelopment will remain a manual, governed process.  
- Enterprise‑wide ModelOps tooling and centralized model registry — beyond current scope; the plan only prepares documentation and reusable components to make later adoption easier.

### Further Considerations

1. Introduce explicit macro‑economic and regulatory environment overlays in calibration monitoring (e.g., adjust expectations for crisis vs. benign periods) to better interpret O/E deviations.  
2. Enhance overrides analysis by linking to staff incentives and performance metrics to detect systematic behavioral biases influencing decisions against the model.  
3. Once stable for PD origination, pilot a small centralized model inventory (even in Excel/SharePoint) to track monitoring status, breaches, and actions across multiple models before investing in a full ModelOps platform.
